# 首次配置向导与闭环引导

## Goal

把仪表盘从占位页升级为 **配置闭环检查清单**：按「启动网关 → 管理鉴权 → 渠道 → 分组 → API 密钥 → 客户端示例」引导用户完成可调用状态，并一键跳转到对应页面。

## Background

- M1 与 API Key 闭环已完成，但用户仍需自行理解多页配置顺序。
- 仪表盘目前仅为占位文案，无法反映当前就绪状态。
- 目标：不新增侧车能力、不强制真上游 Chat 成功；用现有 list API + 壳状态做可读引导。

## Requirements

### R1. 配置检查清单（仪表盘）

展示有序步骤，每步显示 **完成 / 未完成 / 不可检测** 状态：

1. 网关运行中（`gateway.state === running`）
2. 管理 API 已鉴权（`authOk`）
3. 至少一条渠道（`GET /api/v1/channel/list` 长度 > 0）
4. 至少一条分组（`GET /api/v1/group/list` 长度 > 0）
5. 至少一条网关 API Key（`GET /api/v1/apikey/list` 长度 > 0）
6. 客户端对接提示（始终展示 Base URL、`/v1`、`model=分组名`、须用 `sk-octopus-...` 的说明 + curl/SDK 模板）

- 网关未运行或未鉴权时：步骤 3–5 显示为「等待前置条件」并禁止误报失败。
- 每步提供 **去设置 / 去渠道 / 去分组 / 去 API 密钥** 等跳转按钮（由 App 切换导航）。

### R2. 客户端示例块

- 展示可复制的 `curl` 模板（models 与 chat 各一即可）。
- Key 使用占位符 `sk-octopus-YOUR_KEY`，**不**自动把 list 中的完整 Key 塞进模板（避免无意泄露到剪贴板历史以外的展示习惯）；可提示「到 API 密钥页复制明文」。
- 可选：复制 Base URL、`/v1` 单行。

### R3. 刷新与体验

- 进入仪表盘或前置状态变化时刷新 3–5 的检测。
- 提供「刷新检查」按钮。
- 可选：localStorage 记录「折叠详细说明」；**不**强制隐藏清单（可做折叠，默认展开）。

### R4. 文案与一致性

- 全程区分 **管理 JWT** 与 **客户端 API Key**。
- 中文 UI；不出现「任意 api_key 占位即可」。

## Out of Scope

- 多步模态 Wizard（强制逐步点下一步）。
- 自动创建演示渠道/分组/Key。
- 真实上游 Key 的 Chat 200 保证。
- Dashboard 图表 / stats 可视化（可后续任务）。
- 修改侧车二进制。

## Acceptance Criteria

- [x] AC1：仪表盘展示上述步骤 1–6 的状态（1–2 来自壳；3–5 在 running+auth 时 list 检测）。
- [x] AC2：未完成步骤可一键跳转到对应导航页。
- [x] AC3：展示并可复制客户端 curl 模板（含 Base URL 与 Key 占位）。
- [x] AC4：`pnpm lint` / `pnpm build` 通过；不破坏既有冒烟脚本。
- [x] AC5：网关停止时 3–5 不误显示为「已配置成功」。
