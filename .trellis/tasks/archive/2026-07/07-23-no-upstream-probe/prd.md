# 禁止上游供应商测活

## Goal

防止 Model Hub 或开发/AI 流程对**用户配置的上游供应商**发起测活、预热、后台探测类请求，降低账号被风控/封禁风险。将约束写入 code-spec，并核对代码中不存在自动上游探测。

## Requirements

- R1：写入 backend/frontend（及必要 guides）可执行约定：**禁止**对用户供应商（`providers.base_url` + 其 `api_key`）做自动测活、定时 health、预热、空 chat 探测。
- R2：AI/开发联调默认**不**调用用户上游；仅当用户**明确授权某次操作**时，才可对指定供应商发请求。
- R3：产品代码中不得新增「后台 / 启动时 / 定时」上游连通性检查。
- R4：核对并确认当前无后台上游测活；若发现则删除。
- R5：文档/验收文案若暗示「应对上游测活」则改正；保留「本机代理 `/health`」与「真实业务请求上的熔断」说明，且不与「测活」混淆。
- R6：供应商上游 Key 转发与故障转移（**真实用户 Chat 请求**）**保留**。
- R7：**保留**分组页用户主动「拉取模型」（`fetch_provider_models`）：仅点击触发，不算自动测活；不得在打开页面/保存供应商时自动拉取。

## Out of Scope

- 不关闭真实 Chat 请求的故障转移与熔断 HalfOpen 探测位。
- 不删除 OpenRouter 公共榜单（无用户供应商 Key）。
- 不删除本机代理 `GET /health`。
- 不删除 `list_health`（只读熔断内存，不打上游）。

## Acceptance Criteria

- [ ] Spec 明确禁止上游自动测活及 AI 联调默认行为
- [ ] Spec 明确允许：用户点击「拉取模型」、真实 Chat 转发/熔断
- [ ] 代码无启动/定时上游探测；拉取模型仅用户触发
- [ ] 相关文档表述不混淆测活与熔断
- [ ] 门禁通过（若仅 spec/文档变更则 typecheck/lint 或无需构建即可）
