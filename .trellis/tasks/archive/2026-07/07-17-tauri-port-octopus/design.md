# 技术设计：Tauri 移植 octopus（父任务）

## 1. 目标与边界

| 项 | 决策 |
|----|------|
| 路径 | **C 混合渐进**：Tauri 壳 + 可替换网关侧车 → 逐步 Rust 化 |
| MVP | **M1**：启停侧车、无登录管理 UI、SQLite、渠道/分组、OpenAI Chat、基础日志 |
| 平台 | **Windows only** |
| 前端 | **React + Vite + TS + Tailwind** |
| 认证 | 管理台无登录；LLM API **本机免鉴权**；默认 **`127.0.0.1`** |

本设计覆盖 **MVP 架构** 与 **可替换契约**；完整协议矩阵与多 DB 不在 MVP。

---

## 2. 逻辑架构

```
┌─────────────────────────────────────────────────────────┐
│  Tauri App (Windows)                                    │
│  ┌──────────────────┐    invoke     ┌─────────────────┐ │
│  │ React SPA (Vite) │◄─────────────►│ Rust 壳          │ │
│  │ 渠道/分组/日志/设置│              │ paths / gateway  │ │
│  └────────┬─────────┘              └────────┬────────┘ │
│           │ HTTP (127.0.0.1:PORT)           │ spawn    │
│           ▼                                 ▼          │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 网关侧车进程（阶段 1：octopus 兼容二进制/构建产物）   │  │
│  │  · 管理 API（CRUD）  · /v1/chat/completions 转发  │  │
│  │  · SQLite  · 日志/统计（尽力落库）                  │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
         ▲
         │ OpenAI SDK / 其他客户端（免网关 Key）
         └── http://127.0.0.1:PORT/v1/...
```

**原则**：业务状态与转发逻辑在侧车；壳只做生命周期与路径；UI 业务数据优先 HTTP，避免把 CRUD 锁死在 Tauri invoke。

---

## 3. 组件职责

### 3.1 Rust 壳（`src-tauri`）

- 解析并创建 **应用数据目录**（配置、DB、侧车日志、侧车可执行文件缓存）。
- **gateway 状态机**：`idle → starting → running → stopping → idle`；记录 `port`、`pid`、`last_error`。
- 启动侧车：注入 `data_dir`、`host=127.0.0.1`、`port`、日志级别等（CLI 或环境变量，与侧车文档对齐）。
- 健康检查：轮询约定 URL（如 `/health` 或管理 API 探活）；失败则标记 error。
- 退出钩子：优雅停止侧车（信号/子命令），超时后强制结束并 warn。
- 暴露 invoke：`gateway_start` / `gateway_stop` / `gateway_status` / `get_paths`（名称可微调，集中类型声明）。

### 3.2 网关侧车（阶段 1）

- **首选**：官方/自建 **octopus 兼容** 可执行文件，Windows x64，版本钉扎。
- **适配层**（若上游强制 admin 登录或网关 Key）：
  - 管理 UI 无登录：优先找关闭鉴权的配置；否则在侧车前加本地反代/补丁策略（子任务评估，选改动最小者）。
  - LLM 免鉴权：关闭上游 API Key 校验或接受任意 Key。
- 数据与配置落在壳注入的 data dir。
- MVP 功能依赖侧车已有能力：渠道、分组、至少一种负载、Chat 转发、请求日志。

### 3.3 前端 SPA

- 无 auth 路由；布局：侧栏 + 网关状态条 + 页面。
- 页面 MVP：渠道、分组、日志、设置（端口/数据目录展示、侧车状态）；Dashboard 可占位。
- `api/*`：管理 HTTP；`api/tauri.ts`：壳命令。
- 开发模式：Vite dev server + 可选「仅连已启动侧车」便于调试。

## 4. 契约

### 4.1 数据目录（Windows）

- 根：Tauri `app_data_dir`（或等价），例如 `%APPDATA%/<identifier>/`。
- 子路径（建议，实现时写死常量并文档化）：
  - `config/` — 应用级偏好（端口等）
  - `gateway/` — 侧车 config、SQLite、日志
  - `bin/` — 可选缓存的侧车 exe

### 4.2 网络

| 项 | MVP |
|----|-----|
| 默认 host | `127.0.0.1` |
| 默认 port | 可配置，建议 `8080`；占用则失败并提示（或设置页改端口后重启侧车） |
| 管理 API | 侧车原生路径（实现时对照 octopus OpenAPI/路由） |
| 转发 | `/v1/chat/completions`（及侧车已暴露的 models 列表若有） |

### 4.3 鉴权

- 管理 UI：无会话。
- 客户端 → 网关：不强制 Key；SDK 占位 `api_key` 即可。
- 网关 → 上游：使用渠道配置的上游 Key。

---

## 5. 关键流程

### 5.1 应用启动

1. 初始化 paths，确保目录可写。
2. 读本地偏好（端口等）。
3. 自动或手动 `gateway_start`（MVP 建议：**启动应用时自动拉起**）。
4. 前端轮询 status + 侧车 health，就绪后拉渠道列表。

### 5.2 配置渠道并转发

1. UI 创建 OpenAI 兼容渠道（base_url + key）。
2. 创建分组，绑定渠道，负载策略取侧车支持的一种（文档写明，优先 round_robin 或 failover）。
3. 客户端 `base_url=http://127.0.0.1:PORT/v1`，`model=<分组名>`。

### 5.3 应用退出

1. 通知侧车优雅退出（flush 统计）。
2. 等待退出或超时强杀。
3. 再退出 UI 进程。

## 6. 权衡

| 选项 | 选择 | 原因 |
|------|------|------|
| 自研 Rust 网关 vs 侧车 | 先侧车 | 最快对齐功能；契约保留替换空间 |
| UI 全 invoke vs HTTP | HTTP 为主 | 可替换侧车、可浏览器调试 |
| 内嵌 WebView 打侧车 UI vs 自研 SPA | 自研 Vite SPA | 无登录、去 Next、控依赖 |
| 免鉴权 | K1 | 本机工具 + 127.0.0.1 |

**风险**：

1. octopus **AGPL-3.0**：分发二进制/改动源码需合规；文档致谢 + 用户自负。
2. 上游强制登录/Key 与产品 D3/D6 冲突 → 适配成本落在「侧车集成」子任务。
3. 双进程调试复杂度 → 状态机 + 日志目录必须先做好。

---

## 7. 回滚与演进

- **回滚**：侧车版本钉扎；坏版本可换回上一 exe；数据目录保留 SQLite。
- **阶段 2**：Rust 实现同源 HTTP 契约，壳只换启动目标；UI 尽量零改。
- **功能扩展**：协议/价格/多 DB 按子任务叠加，不破坏 M1 契约。

---

## 8. 子任务切分（父任务地图）

| 顺序 | 子任务 | 可独立验收 |
|------|--------|------------|
| 1 | 脚手架：Tauri + Vite React + 数据目录 + 空壳窗口 | Windows 可 `tauri dev` 出窗口 |
| 2 | 侧车集成：获取/钉扎二进制、启停、health、优雅退出 | AC6 + 状态条 |
| 3 | 管理 UI MVP + 侧车 API 适配（无登录） | AC1、渠道/分组/日志页 |
| 4 | E2E 与文档：Chat 转发、客户端说明、AC2–AC5 | M1 清单勾选 |

父任务本身不写业务代码，负责集成验收与 PRD 闭环。
