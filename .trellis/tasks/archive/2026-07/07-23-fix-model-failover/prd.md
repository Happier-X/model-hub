# 简化故障转移：取消熔断与自动开关

## 目标

取消供应商级熔断与分组 `auto_failover` 开关。每个请求始终从分组队列中第一个启用候选项开始，当前候选项出现任意错误且响应尚未提交客户端时，按顺序故障转移到下一候选项，直到成功或队列耗尽。

## 背景与已确认事实

- 用户实际收到错误体：`{"error":"当前 API 不支持所选模型 gpt-5.6-sol","type":"error"}`，但故障转移未发生。
- 当前故障转移实现位于 `src-tauri/src/proxy/forward.rs`；仅将部分状态码和网络错误视为可换源，普通 `400`、`404` 会立即返回。
- 现有熔断实现位于 `src-tauri/src/proxy/circuit.rs`，通过 `CircuitRegistry` 参与代理运行时、路由、流式回调和健康展示。
- `list_health`、`HealthBadge.vue` 及供应商/分组页面中的健康状态仅展示进程内熔断状态，不会探测上游。
- 分组表、领域模型、IPC 类型和 `GroupsPage` 存在 `auto_failover`；关闭时仅尝试第一候选项。
- 分组候选项已有稳定队列顺序；禁用供应商当前会被跳过。
- 流式首个数据块提交后禁止拼接其他上游；本任务不改变该约束。

## 已确认产品决策

- **取消供应商熔断**：不保留失败计数、Open/HalfOpen 状态或探测位；历史请求结果不影响下一请求的起点。
- **彻底删除 `auto_failover`**：移除数据库字段、后端领域与 IPC 契约、前端开关和相关分支；故障转移始终开启。
- **所有错误均触发换源**：响应提交前，网络错误、超时、任意非 `2xx` 状态（包括普通参数 `400`、`404`）、明确的 `2xx` 结构化错误响应都继续下一候选项。
- **删除熔断健康展示**：移除 `list_health`、健康快照类型、健康徽章和页面引用；本机代理自检 `GET /health` 与供应商熔断无关，继续保留。

## 需求

- R1：每个请求始终从分组内第一个启用候选项开始，按现有队列顺序尝试，不得受历史成功或失败影响。
- R2：响应尚未提交客户端时，当前候选项的任意失败都必须继续下一启用候选项，包括网络错误、超时、任意非 `2xx` HTTP 响应和明确的 `2xx` 结构化错误响应。
- R3：结构化错误识别应兼容字符串形式的 `error`、对象形式的 `error.message`、顶层 `message` 等常见错误信封；正常 `2xx` 成功 JSON 和正常 SSE 数据不得误判。
- R4：非流式响应与流式首包提交前遵循一致的“错误即换源”原则；流式首包一旦提交，后续静默超时或读取错误只终止当前流并记录日志，禁止换源拼接。
- R5：队列耗尽时，如果最后一次失败包含上游 HTTP 响应，客户端收到该最后候选项的原始状态、响应头和错误体；如果最后一次失败是网络或超时等无上游响应的错误，则返回明确的网关错误。
- R6：故障转移与最终错误日志不得包含 API Key 或完整消息内容；保留状态码、候选项、截断错误摘要及 `failover_from` / `failover_to` / `failover_reason`。
- R7：彻底删除供应商熔断状态机及其运行时依赖，不再记录连续失败、跳过供应商或占用 HalfOpen 探测位。
- R8：彻底删除 `auto_failover` 的新库 schema、旧库迁移后 schema、领域对象、创建/更新 payload、前端类型和 UI；迁移不得丢失已有分组、分组条目及其他数据。
- R9：移除仅服务于熔断展示的 `list_health` 命令、健康快照、健康徽章与页面调用；不得新增替代性的后台测活。
- R10：保持现有候选项排序、供应商启用状态、超时常量、本机 `GET /health` 和流式首包提交后不换源的规则不变。

## 验收标准

- [ ] AC1：第一候选项返回“不支持所选模型”的 `400` 错误、第二候选项成功时，最终请求成功且两个候选项按顺序各被访问一次。
- [ ] AC2：第一候选项返回普通参数 `400`、`404` 或其他任意非 `2xx`，第二候选项成功时均发生故障转移。
- [ ] AC3：第一候选项发生网络错误或首包前超时、第二候选项成功时发生故障转移。
- [ ] AC4：明确的 `2xx` JSON 错误信封在非流式及可安全判断的流式首包前触发换源；正常 `2xx` 成功响应不换源。
- [ ] AC5：连续多次请求时，每次都从队列第一个启用候选项开始，不存在熔断跳过、失败计数或 HalfOpen 并发限制。
- [ ] AC6：全部候选项返回 HTTP 错误时，客户端收到最后一个候选项的原始状态、响应头与错误体；最后一次为无响应错误时收到明确网关错误。
- [ ] AC7：新建数据库不含 `groups.auto_failover`；已有数据库升级后移除该列并完整保留分组及条目数据。
- [ ] AC8：创建、更新、查询分组及前端 UI 不再包含 `auto_failover`；供应商和分组页面不再调用或展示熔断健康状态。
- [ ] AC9：流式首包提交后发生静默超时或读取错误时不尝试下一候选项，并只写一条最终日志。
- [ ] AC10：Rust 格式检查、测试、静态检查，以及前端 lint、单测、类型检查和构建全部通过。

## 范围外

- 不改变流式首包提交后的换源限制。
- 不新增后台测活、模型预探测、预热或额外上游请求。
- 不调整候选项排序算法、权重或模型能力重排。
- 不改变供应商启用/禁用功能。
- 不删除本机代理自检 `GET /health`。

## 技术约束

- `2xx` 错误识别必须基于明确的结构化错误信封，不能仅凭任意正文关键词。
- 错误摘要继续遵循现有长度限制与脱敏要求。
- SQLite 迁移删除 `auto_failover` 时必须使用兼容当前 SQLite 能力且可测试的数据保留方案。
- 删除熔断后不得遗留无调用的状态机、无效命令注册或前端死代码。
